## 摘要

为 compound-engineering 插件内部的虚拟团队增加每日站会提醒功能：固定时间点自动触发提醒，成员通过编辑共享的 Markdown 文件提交站会内容（以 `@用户名` 为标题、标准三段式格式），最终自动生成一份新的站会汇总 Markdown 文件。

## 问题陈述

团队目前缺乏内置的轻量每日同步机制。成员的每日进展、计划和阻塞信息零散分布，缺少结构化的收集与汇总流程，导致信息聚合成本较高且不透明。

## 关键决策

- 团队模型采用插件内部虚拟团队，不依赖外部协作工具。
- 站会提醒通过固定时间点自动触发。
- 成员通过编辑共享 Markdown 文件提交内容，降低交互成本。
- 成员标识采用 `@用户名` 标题格式。
- 站会内容采用标准三段式模板：昨天做了什么 / 今天计划什么 / Blocker。
- 汇总产物为一份新的、独立的 Markdown 文件，不直接修改输入文件。

## 参与者

- **站会召回器**：到达固定时间点时自动触发提醒流程。
- **团队成员**：在共享文件中编辑自己区块的站会内容。
- **汇总引擎**：读取共享输入文件并生成汇总 Markdown 文件。

## 核心需求

| ID  | 需求                                                                          |
| --- | ----------------------------------------------------------------------------- |
| R1  | 插件必须支持在固定时间点每日自动触发站会提醒。                                |
| R2  | 站会提醒必须指向一个可编辑的共享 Markdown 输入文件。                          |
| R3  | 团队成员必须以 `@用户名` 作为区块标题来提交自己的内容。                       |
| R4  | 每个成员的内容必须遵循标准三段式结构：昨天做了什么 / 今天计划什么 / Blocker。 |
| R5  | 汇总引擎必须能够读取输入文件并解析所有已提交的 `@用户名` 区块。               |
| R6  | 系统必须生成一份新的站会汇总 Markdown 文件，内含所有成员的三段式内容。        |
| R7  | 汇总文件必须包含站会日期与成员列表。                                          |
| R8  | 汇总文件的输出路径必须可配置，且默认值与输入文件位于同一目录树。              |

## 关键流程

### 流程一：定时触发提醒

1. 到达每日预设固定时间点。
2. 插件生成/更新共享站会输入文件模板（含当日日期和空的三段式区块）。
3. 向团队成员发出提醒（CLI 输出或可配置的通知渠道）。
4. 进入等待收集状态。

### 流程二：成员提交内容

1. 成员打开共享输入文件。
2. 添加或更新自己的 `@用户名` 标题下的三段式内容。
3. 保存文件。

### 流程三：生成汇总

1. 触发汇总逻辑（按固定时间或手动触发）。
2. 读取共享输入文件，解析所有 `@用户名` 区块及三段式内容。
3. 生成汇总 Markdown 文件，按成员分组展示昨日完成、今日计划、阻塞信息。
4. 归档/展示汇总文件路径。

## 验收示例

- **场景一**：固定时间点到达时，CLI 输出类似 ` Daily Standup Reminder: 请填写 docs/standups/input-2026-06-06.md`。
- **场景二**：Alice 在输入文件中添加 `@alice` 区块并填写三段式内容，保存后汇总文件正确包含她的内容。
- **场景三**：汇总文件按 `@用户名` 分组，每组下均有“昨日完成”“今日计划”“阻塞”三个小节，空值视为未提交而非报错。

## 成功标准

- 每日固定时间点能可靠触发提醒。
- 成员提交内容后，汇总文件能正确反映所有已提交成员的三段式内容。
- 未提交成员在汇总文件中以“未提交”或留空方式体现，不阻塞汇总流程。
- 汇总文件格式清晰、可直接阅读，无需额外工具解析。

## 范围边界

### 包含

- 固定时间点自动提醒。
- 共享 Markdown 输入文件的模板生成与维护。
- 标准三段式内容的解析与汇总。
- 汇总 Markdown 文件的新增输出。

### 不包含

- 外部协作工具集成（如 Slack、飞书、企业微信）。
- 实时多人协作编辑。
- 多轮迭代或评论线程。
- 成员权限或角色管理。

## 依赖与假设

- 插件具备定时任务能力，或可通过外部调度器（如 cron、launchd）配合 CLI 命令触发流程。
- 团队成员列表可通过配置文件维护。
- 共享文件的路径可配置。
- 所有成员遵循 `@用户名` + 标准三段式的提交约定。

## 待确认问题

- 站会汇总的触发时间点如何定义？（输入文件的截止时间 vs. 汇总生成时间）
- 汇总文件的命名规范与存放目录是什么？
- 超时未提交的成员在汇总中如何处理？（记为“未提交”还是继续等待？）
- 提醒的具体通知渠道是什么？（仅 CLI 输出、系统通知，还是可插拔？）
- 历史站会记录是否需要保留？保留多长时间？
- 团队成员列表如何配置？（静态文件、插件配置，还是动态获取？）
- 是否需要支持手动触发单次站会或补录汇总？
