Project Decomposition & Orchestration Skills

Seven ranked directions for new ce- skills that take a big project, fan it into tracked tasks, map their dependencies, and run them through the pipeline with the human in the loop only where judgment is required.

repo-grounded 2026-06-21 focus: project-orchestration-skills 38 raw → 7 ranked · 11 cut 5 axes

The shape of the answer

The grounding made one thing clear: most of the machinery the user's workflow needs already exists inside the plugin. ce-plan decomposes a single feature into stable U-ID units with dependency citations. ce-work already fans out parallel, worktree-isolated sub-agents with file-collision detection and dependency-order merge. lfg already runs the full autonomous pipeline for one unit. The gap is not execution primitives — it is the layer above them: nothing decomposes a multi-feature project, nothing carries per-task routing metadata, nothing maps dependencies across tickets, and nothing tracks where the whole project stands.

So the strongest framing — surfaced independently by four of five ideation frames and confirmed in verification — is not one mega-orchestrator skill but a small family of thin, composable skills sharing one durable artifact: a committed task-graph the repo owns. Each skill reads and writes that graph; state lives outside any single Opus session, so any session (or a cron, or a second machine) can resume by reading a file. That reframe is what makes the autonomous-execution vision robust instead of fragile.

Grounding Context

What already exists (build above, do not rebuild)

  • ce-plan decomposes one feature into stable U-ID units carrying Goal, Requirements, Dependencies (cited by U-ID), Files, Approach, Execution note, Test scenarios, Verification — and can create a Linear/GitHub issue from a plan. Tracker is read from AGENTS.md project_tracker:.
  • ce-work already dispatches parallel sub-agents with isolation: "worktree" + run_in_background: true, does pre-dispatch file-collision detection, merges in dependency order (aborting + re-dispatching on conflict), and cleans up worktrees.
  • lfg already runs the full single-unit pipeline: plan → work → simplify → review → fix → residual-filing → test → commit → push → PR → CI-watch (3-attempt repair), with hard HITL gates and an autopilot residual-filing path.
  • ce-code-review fans out ~16 reviewer personas in parallel via a bounded scheduler with backpressure. ce-worktree is a detect → native-tool → git-fallback state machine that never nests.

Confirmed gaps (what the new skills fill)

  • No cross-feature / project-level decomposition above ce-plan.
  • No per-task next-CE-stage tag and no per-task model-tier recommendation — the tier vocabulary exists but is agent-side only, never embedded in a unit.
  • No cross-ticket dependency map, no orchestrator that tracks project state.
  • Linear is write-once: issues are created but never read back, status-synced, or linked as sub-issues/dependencies.

Load-bearing learnings & prior art

  • Fan-out has a real crossover (~5–7 units before delegation pays off; batch ~5, never loop per-unit). Pass paths, not content, to sub-agents.
  • Gate autonomy with a visible preview, not a single keystroke — a prior LFG-autopilot rubber-stamp failure is documented in ce-pipeline-end-to-end-learnings.md.
  • Model git/worktree lifecycle as an explicit state machine; keep always-on menu routing inline in SKILL.md.
  • External: Spec-Kit's Constitution→Specify→Plan→Tasks→Implement; Graph-Harness DAG scheduling (named failure modes: missing/spurious deps, over/under-decomposition); issue-tracker-as-coordination-bus (Linear Agent API 2026); ~40% of multi-agent pilots fail on context/recovery gaps, not model capability.

1

ce-decompose — the project-decomposition skill above ce-plan

Axis 1 · DecompositionConfidence 85%Complexity Low

A new skill that sits one level above ce-plan: it takes a big project description (or a brainstorm doc) and fans it into a set of feature-sized task stubs, reusing ce-plan's own unit schema verbatim so every emitted task already carries the fields a downstream executor consumes — Goal, Files, Dependencies (cited by stable ID), Test scenarios, Verification. It emits one index plus per-task plan fragments, with IDs globally unique across the whole project rather than per-feature. Two cheap, high-value capabilities fold in: a critical-path / slack pass over the dependency DAG (which tasks are schedule-critical) and a granularity guard that audits its own output for missing dependencies (a task reads a file no prior task creates), spurious edges, and over/under-decomposition before emitting.

Basis
direct: the per-unit field contract and "cite by U-ID" dependency format already exist in ce-plan (evidence-decomposition-routing.md); the grounding confirms there is no cross-feature decomposition today. external: Spec-Kit's epic→plan→tasks hierarchy; Graph-Harness names the exact decomposition failure modes the guard checks for.
Rationale
This is the precise friction the user described — they currently run a manual Opus session to shatter a project into features and hand-create tickets. Reusing the existing unit schema means ce-work / lfg need zero new input format, and critical-path tells the human where attention actually matters. It is the entry point the rest of the family depends on.
Downsides
Decomposition quality is the whole ballgame; a bad split produces a wrong dependency graph that silently mis-schedules everything downstream. The granularity guard mitigates but does not eliminate this. Large projects may over-decompose into too many thin tasks — needs a sensible floor.
2

The committed task-graph as coordination bus (+ stateless re-orient)

Axis 4 · OrchestrationConfidence 82%Complexity Medium

Make the durable artifact a version-controlled task-graph the repo owns (e.g. docs/plans/<project>/task-graph): nodes are tasks with status, next-stage, model tier, and tracker URL; edges are the dependency citations ce-decompose already produces. The git repo — not a single Opus session and not an external tracker — is the source of truth; Linear becomes a downstream projection. Paired with this is a stateless re-orient operation that holds nothing in memory: on every call it reconstructs each task's true state by cross-referencing the graph file, git (gh pr view, branch/merge state), and the tracker, so a fresh or context-compacted session produces identical output.

Basis
direct: ce-work already insists "progress lives in git commits and the task tracker, not the plan body" — a committed graph file is the natural home for that progress; gh pr view and Linear get_issue_status are already used/available. reasoned: an orchestrator that holds state in-context dies at compaction and can't run from cron or a second machine; a pure function of (file + git + tracker) is the precondition for resumability.
Rationale
This is the architectural spine that makes every other idea coherent and cheap — ce-next, the recovery manifest, and tracker-sync all become thin operations once a durable graph exists. It directly answers the user's "let it know where things are" requirement by making "where things are" a file anyone can read, not a memory in one chat.
Downsides
Introduces a second representation alongside the tracker — the file/tracker drift question must be answered decisively (the proposal: file is canonical, tracker is a mirror). Schema design needs care to stay diffable and reviewable in PRs. Reconstruction cost grows with project size.
3

Per-task ce_stage + model_tier routing fields

Axis 2 · Routing annotationConfidence 88%Complexity Low

Add two optional fields to every task/unit: ce_stage (the next pipeline stage — brainstorm / plan / work / lfg) and model_tier (the recommended model, drawn from the existing extraction/generation/ceiling vocabulary). They are inert metadata until an orchestrator reads them, so adding them is zero-risk and additive to the current schema. A thin computed pass (ce-route, or a deepening phase inside ce-plan) stamps them by signal — open requirements route to brainstorm; a fully-specified, low-complexity task routes straight to lfg; a cross-cutting/architectural task routes to plan and recommends the ceiling tier.

Basis
direct: the dossier names these exact absences — "No per-unit next-CE-stage tag," "No per-unit model recommendation field," "tiering is agent-side only." The existing execution: knowledge-work field is precedent for downstream-consumed routing metadata on a unit. external: code-health-based model routing (Triage).
Rationale
This is the connective tissue between decomposition and autonomous execution and the highest confidence / lowest cost item in the set. Without a carried stage tag, the orchestrator must infer what to do with each task; with it, dispatch is a table read, not an LLM judgment. It is exactly the metadata the user already hand-writes into tickets today.
Downsides
The stamping rules are genuinely heuristic — the verifier flagged that "open requirements → brainstorm" is invented, not grounded. The rules must be calibrated and treated as a recommendation a human can override, not a hard gate. Risk of the fields drifting stale if not recomputed as tasks evolve.
4

ce-next — the ready-set recommender

Axis 4 · OrchestrationConfidence 84%Complexity Medium

An orchestrator skill that reads the task-graph, computes the dependency frontier (every task whose dependencies are all satisfied), checks live status, and — instead of dumping a menu of N ready tasks — recommends one highest-leverage next move with a one-line rationale and the exact command to run it (/lfg T7, or /ce-brainstorm T3 when its stage is still brainstorm). It detects the OS-scheduler pathologies the DAG makes visible: deadlock (a dependency cycle), and starvation (a task ready across many cycles but never picked). Recomputed live each call, so decomposition is continuous, not a one-time upfront act.

Basis
direct: the dependency edges already exist as U-ID citations; frontier computation is mechanical over them. external: Graph-Harness DAG ready-set scheduling; the HITL principle that reversible actions shouldn't cost a decision. reasoned: a dependency map that still makes the human scan N ready tasks has only moved the cognitive load, not removed it.
Rationale
This is the user's "tell me what to do next" requirement made concrete and judgment-free for the mechanical part. Recommending one move with a rationale is the difference between a dashboard and an orchestrator.
Downsides
"Highest-leverage" ranking is a heuristic that can mislead if mis-calibrated; the human must be able to override the single recommendation. Cycle/starvation detection adds value only once enough of the family exists to produce real graphs.
5

ce-fanout — autonomous worktree /lfg executor with visible gates

Axis 5 · Autonomous executionConfidence 80%Complexity High

The autonomous executor the user wants — assembled from existing primitives rather than rebuilt. It pulls the ready set, lifts ce-work's file-collision check to cross-ticket scope to partition ready tasks into collision-free parallel batches (bounded by the ~5–7 fan-out crossover), shows a visible preview of the planned batch (which tasks, which worktrees, which model each), and on confirmation spawns one worktree-isolated sub-agent per task each running /lfg end-to-end. Two safety mechanisms fold in: a circuit breaker that halts the batch after K consecutive failures (escalating to the human instead of burning the whole run), and consolidated HITL — gate decisions arising mid-wave are collected into one batched checkpoint per wave rather than M gates × N tasks inline.

Basis
direct: ce-work already does worktree-isolated parallel dispatch, the cross-ticket-able file-collision check, and dependency-order merge; ce-code-review already has the bounded backpressure scheduler; lfg already is the per-unit pipeline. external: the documented LFG-autopilot rubber-stamp failure motivates the visible gate; circuit breakers are standard distributed-systems prior art.
Rationale
This is the headline payoff — walk away, return to N tickets shipped as PRs. Because it composes shipped primitives, the new surface is the drain-the-batch loop and the gates, not new execution machinery. The visible preview + consolidated checkpoint are the explicit antidote to the known rubber-stamp anti-pattern.
Downsides
Highest complexity and highest blast radius in the set; unbounded fan-out of an autonomous pipeline is exactly where "invisible state" and runaway cost bite. Code isolation (worktrees) is not runtime isolation — port collisions and shared DB/`.env` state across parallel /lfg runs are a real, unsolved failure mode that must be scoped. Should not be built before ideas 1, 2, and 7 exist.
6

ce-tracker-sync — make Linear a live coordination bus

Axis 3 · Linear integrationConfidence 78%Complexity Medium

Close the write-once gap. Build the safe direction first: a pull that reads Linear issue status into the task-graph and surfaces a diff for the human to confirm (catching tickets a developer closed manually outside CE). Then add the write side: create one sub-issue per task with parent/child and dependency relations, and write status transitions back as tasks move (In Progress on dispatch, Done on merged PR, a comment on failure). Invoked as a hook from ce-decompose / lfg, never something the human has to remember to call.

Basis
direct: evidence-linear.md confirms every gap — "No status tracking or sync," "No sub-issues or dependency links," "ends after URL display." The Linear MCP tools (save_issue, get_issue_status, save_comment) are available, and the project_tracker: linear detection + MCP-preferred/CLI fallback chain already exist. external: issue-tracker-as-coordination-bus; Linear Agent API 2026 (agents as workspace members assigned issues).
Rationale
The orchestrator's ready-set is only as good as its view of status; without read-back, CE must trust git history alone and goes blind to human-driven changes. Pull-first is independently useful and de-risks the write side. This is what turns the tracker from a write-only dumping ground into the live board the user wants.
Downsides
Two-way sync invites the canonical-source question (settled by idea 2: file is canonical). Status-mapping (CE stage → Linear state) is an integration contract someone must own. Depends on Linear MCP availability, which can be absent in headless/cron runs — needs graceful degradation.
7

Cross-session recovery manifest

Axis 5 · Autonomous executionConfidence 81%Complexity Medium

A durable checkpoint written at every batch boundary recording, per in-flight task: status, the absolute worktree path and branch, whether a PR is open, and the last clean merge. If a session dies or context compacts mid-run, recovery becomes a read operation — the skill reconstructs exactly which worktrees exist, which PRs are open, and where to resume — instead of a manual detective exercise across orphaned branches.

Basis
direct: ce-work's cleanup sequence already records worktree path and branch — the same data, written durably; ce-worktree's detection logic can re-discover existing worktrees on resume. external: ~40% of multi-agent pilots fail on context-management / recovery gaps, named as the primary failure mode, not model capability.
Rationale
Without recovery, a 10-task autonomous run that dies at task 7 leaves the project in an unknown state — some worktrees gone, some PRs open, some branches orphaned. The manifest is the precondition that makes ce-fanout trustworthy enough to actually walk away from.
Downsides
A persistent checkpoint can itself drift from reality if not reconciled against git on read (mitigated by treating git as authoritative and the manifest as a hint). Choosing the durability tier matters — must survive a session but stay out of the plan body.

How they sequence

Suggested build order

Foundation first: ideas 1 (ce-decompose) and 2 (task-graph) ship together — decomposition has nowhere to write without the graph, and the graph has nothing to hold without decomposition. 3 (routing fields) is the cheapest add and rides along. 4 (ce-next) then turns the graph into guidance for a human running tasks by hand — a complete, valuable product on its own, no autonomy required. 6 (tracker-sync, pull-first) and 7 (recovery manifest) are the durability layer that must precede 5 (ce-fanout), the highest-risk capstone. This lets you ship value at every step and only reach full hands-off execution once the state and recovery substrate exists.
