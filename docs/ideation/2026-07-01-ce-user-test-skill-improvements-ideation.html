Grounding Context

Codebase context. The suite is four skills on branch feat/user-test-skill-v2: ce-user-test (~400-line SKILL.md plus 12 reference files), ce-user-test-iterate, ce-user-test-commit, ce-user-test-eval. The agent drives a visible Chrome window via the claude-in-chrome MCP, scores areas 1-5 (UX plus optional output quality), and persists compounding state in tests/user-flows/: a maturity ladder (Uncharted → Proven → Known-bug), regression probes, queries, journeys, novelty fingerprints, and run history across 8 artifact files at schema v10.

Evidence highlights. Five evidence dossiers (671 lines, file:line-cited) surfaced: a v1→v10 migration ladder living as cascading prose conditionals; six rotation caps specified independently in six files; atomic writes documented for one of eight mutated files; verification failures that block promotion but never demote; a scoring rubric claimed "absolute" with no calibration mechanism; disconnects trending 6→10 while sibling skills (ce-test-browser, ce-dogfood) mandate the agent-browser CLI exclusively; and a self-eval whose spec names its own missing Eval 4.

Past learnings applied. Agent judgment for quality/maturity state but explicit state machines for externally-verifiable state; anchored discrete rubrics over floats; script-first architecture for mechanical data processing (60-75% token cut); paired old-vs-new blind injection for validating skill prose; monolith-split anti-patterns (prose budgets creep without deterministic gates).

External research. GUITester (arXiv 2601.04500) names Goal-Oriented Masking and Execution-Bias Attribution as agent-tester failure modes (+15.5pp F1 from decoupled roles). LLM-judge literature validates the 1-5 scale but flags self-enhancement bias and rubric conflation (Nielsen severity vs SUS-style experience). 2026 QA-market scan: no funded competitor documents cross-run compounding memory — the maturity ladder is genuine differentiation. UXBench: ground UX scores in interaction evidence.

1

Deterministic core refactor — scripts own the arithmetic, the agent keeps the judgment

compounding-state-schemaConfidence 85%Complexity Medium5-frame convergence
Description
One refactor bundling the four mechanical subsystems that today live as LLM-followed prose: (a) a bundled scripts/migrate-test-file (SKILL_DIR-anchored) replaces the nine cascading v1→v10 migration conditionals — a schema bump becomes a table row, not a new paragraph; (b) a central cap/threshold registry replaces six independently-specified rotation caps (10/20/10/10/50/6mo) plus scattered thresholds (7-vs-5 pattern surfacing, -0.5 delta, restart 15); (c) a write-ahead commit journal makes the 11-step, 8-file commit resumable and all-or-nothing instead of atomic-for-one-file-only; (d) the undefined "semantic title search" issue-dedup fallback becomes the 70% word-overlap algorithm the skill already specifies for probes. A synthetic aging harness (simulate 100 commit cycles, assert caps hold and IDs stay unique) becomes the regression gate. Scores, maturity calls, and notes stay agent-owned — only arithmetic and file mutation move to code.
Basis
direct: SKILL.md:45-51 (nine migration blocks, each "Do NOT rewrite on read", contradicted by commit mode force-upgrading to v10 at SKILL.md:323); six cap citations across five files; SKILL.md:321-323 atomic write for one of eight artifacts; SKILL.md:350-357 unspecified dedup fallback vs probes.md's 70% overlap precedent. Reinforced by the repo's own script-first learning (60-75% token cut) and the deepen-plan v4.1 consolidation precedent.
Rationale
This is where five independent ideation frames converged. Every future schema version, cap change, and commit step currently adds prose an LLM must re-execute correctly on every run — the failure class (drift, miscounts, partial-commit corruption) compounds with the state it manages. Scripting it is the repo's own documented playbook applied to its clearest unexploited instance.
Downsides
Adds bundled-script infrastructure (cross-platform Python/Node choice, SKILL_DIR anchoring, permission prompts) to a currently pure-prose skill; the verifier cautioned that the maturity-transition portion must stay agent-owned or it contradicts the founding agent-judgment learning.
2

Browser substrate strategy — tool-neutral adapter, agent-browser default, mid-run failover

browser-execution-resilienceConfidence 80%Complexity High4-frame convergence
Description
Rewrite Phases 2-3 in tool-neutral action verbs (navigate, read-page, evaluate, screenshot) with an adapter table mapping verbs to whichever engine is present: claude-in-chrome when the user wants the visible, shared-login window; agent-browser CLI otherwise. On disconnect_counter ≥ 3 or WSL detection, divert remaining areas to the CLI engine instead of aborting, recording which engine scored each area. Add a replay-before-blame attribution step: on any tool failure, run a known-good minimal action first — success means the failure was the app (score it, probe it); failure means the connection (recovery path).
Basis
direct: SKILL.md:3 hardcodes claude-in-chrome in the frontmatter; ce-test-browser/SKILL.md:18 and ce-dogfood mandate agent-browser exclusively; disconnects trending 6→10 (SKILL.md:252); mcp__claude-in-chrome__* names violate the plugin's cross-platform rule (Claude-Code-only in a suite shipped to Codex/Cursor/Gemini). external: GUITester's Execution-Bias Attribution — the current recovery path treats every failure as transient, so real app defects get retried and absorbed into the disconnect counter.
Rationale
The skill's most-observed failure mode becomes run-degrading instead of run-killing; the WSL abort disappears; the suite stops being the only sibling on the fragile substrate; and mis-attributed failures stop corrupting both the resilience counters and the scores at once.
Downsides
Largest change surface of the seven; the visible-window/shared-login experience is part of the skill's identity and must remain a first-class mode, not a legacy path; per-engine behavioral differences could add scoring variance across engines.
3

Calibration deck + blind proficiency seeds — make "scores are absolute" a measured property

scoring-maturityConfidence 75%Complexity Medium4-frame convergence
Description
Ship a small frozen deck: 3-10 artifacts (reports and evidence bundles, including deliberately ambiguous cases) with locked reference scores. Each run — or each eval, as an "Eval 0" — re-scores the deck with the current model and skill version, records deviation, and flags the run's scores as uncalibrated when any anchor drifts beyond ±1. Alongside it, 3-6 known-bad seed artifacts (a P1 buried in DETAILS, a missing Proven-regression marker) that the grader must FAIL — proving the evals can still fail at all, since the current graduation rule rewards consecutive passes with no vacuousness check.
Basis
direct: SKILL.md:178 "Scores are absolute per this rubric" with zero variance mechanism (flagged CRITICAL GAP in the evidence dossier); ce-user-test-eval/SKILL.md:170-173 treats consecutive passes as pure health signal. external: clinical-lab QC (control samples per run, CAP blind proficiency testing) and mutation testing; the repo's safe-auto-rubric-calibration learning (measure variance on ambiguous cases).
Rationale
Every trend, delta, promotion gate, and adversarial trigger assumes score comparability across sessions and models — today that assumption is unfalsifiable. This turns the suite's weakest epistemic claim into its most defensible one using artifacts the pipeline already produces.
Downsides
Deck curation is real work and the deck itself can go stale as the rubric evolves; adds per-run cost (a few extra scoring calls); reference scores must themselves be argued once.
4

Anomaly ledger + Eval 4 — defeat Goal-Oriented Masking with a mandatory side-channel

self-eval-loopConfidence 80%Complexity Low
Description
Require the exploring agent to append to .user-test-anomalies.jsonl at every area transition — one line per anomaly noticed en route (or an explicit none), with area and evidence pointer. Report generation must consume the ledger, making silence distinguishable from vigilance. Then implement the already-scoped Eval 4 (the eval spec itself says "candidate for a future Eval 4" for verification-mismatch surfacing) as a ledger→report coverage check: nothing noticed may vanish between Phase 3 and Phase 4.
Basis
external: GUITester's Goal-Oriented Masking (agents prioritize task completion over reporting anomalies noticed en route) and NASA ASRS's separate non-punitive reporting channel. direct: SKILL.md:226 "Sections with no items are omitted" — the dispatch format actively rewards silence with a cleaner-looking report; ce-user-test-eval/SKILL.md:73-75 names the missing eval.
Rationale
Masked anomalies are the findings exploratory testing exists to catch, and the current format structurally rewards masking. One schema field, one report section, and one eval close a published failure mode — the cheapest high-value change in the set.
Downsides
Mandatory per-transition entries add small overhead and can degenerate into rote none entries; Eval 4's cross-artifact matching needs the same semantic-matching care Eval 3 already requires.
5

Evidence array per scored area — every score cites what it saw

reporting-commitConfidence 78%Complexity Low
Description
Mandate an evidence array per scored area in .user-test-last-run.json — action references, DOM reads, timings, screenshot paths — and render score→evidence links in the report. An unevidenced "4/5" becomes structurally impossible rather than merely discouraged.
Basis
direct: verification-patterns.md:18 already records exact counts ("2 of 8 sampled results had mismatched condition badges") — the precedent exists for verification but not for UX/Quality scores. external: UXBench (arXiv 2606.16262): grounding UX scores in interaction evidence is what makes them comparable across runs.
Rationale
This is the enabling artifact for the rest of the scoring program: the calibration deck (idea 3) freezes evidence bundles, Eval 4 (idea 4) checks coverage against evidence, and any future scorer separation would grade from it. Sequencing it early makes three other directions cheaper.
Downsides
Schema bump (v11) and slightly heavier run artifacts; evidence quality itself is agent-produced and needs a light format contract to stay useful.
6

Spaced-repetition probe scheduler — one mechanism replaces budgets, caps, and ordering rules

compounding-state-schemaConfidence 60%Complexity Medium
Description
Give every probe, query, and journey a next_due_run computed from its pass streak (SM-2 style: interval grows on pass, resets on fail); a run executes whatever is due. The tiered Proven budget table, the rotation caps, and the probe-ordering combinator collapse into one scheduling formula. Nothing ever leaves the deck — mature items just come due less often, closing the "Proven rot" hole structurally (under the current floor, a long-Proven area gets one call forever).
Basis
direct: probes.md:64-70 — the confidence field is updated like an ease factor but never affects scheduling; rotation caps are inconsistent at 10/20/50 across files. external: SM-2 (SuperMemo/Anki) — interval growth with lapse reset is the canonical solution to cheap maintenance of a growing set of known facts.
Rationale
A rare simplify-and-strengthen trade: it deletes three scattered subsystems (a flagged drift source) while making re-testing coverage principled instead of budget-starved. Pairs naturally with idea 1's script layer, which would own the interval arithmetic.
Downsides
A real redesign of run-targeting semantics, not a patch; interval parameters need tuning against actual run cadence; medium confidence because the maturity ladder's current behavior is proven in use and this changes its rhythm.
7

Two-channel rubric — defect severity per finding, experience score per area

scoring-maturityConfidence 65%Complexity Medium
Description
Split the single 1-5 scale that currently spans "Broken — cannot complete the task" to "Delightful": findings get a discrete Nielsen-style severity (cosmetic / minor / major / catastrophic); areas keep an experience score. Promotion gates read severity (any major+ blocks); trends read experience. The undefined judgment calls scattered through the maturity rules — "minor CSS issues do not trigger demotion", "a cosmetic issue does not warrant demotion" — get a concrete home instead of living as ad-hoc exceptions.
Basis
direct: SKILL.md:170-176 (one axis from broken to delightful) and SKILL.md:313 (undefined "minor CSS" exception). external: Nielsen's 4-point severity scale vs SUS-style experience scales are distinct rubric traditions; LLM-judge literature flags conflated rubrics as a judge-noise source.
Rationale
Half the fragile maturity rules are workarounds for severity being smuggled inside an experience number. Separating the channels reduces judge noise at the source and makes the promotion gate mean something specific.
Downsides
Schema and report changes ripple through promotion, demotion, bug filing, and both evals; historical score series become only partially comparable across the transition.
