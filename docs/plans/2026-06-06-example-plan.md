# IPv6 支援拡張計画

## Summary

本計画は、プロジェクトをフル IPv6 ネットワーク環境に展開できるよう支援する一連の変更を提案する。対象範囲は OS ネットワークスタックの設定、標準的なインフラスクリプトの拡張、リバースプロキシ設定、ファイアウォール規則、テスト項目、ドキュメント整備に及び、単一システムまたは小規模ネットワーク両方のユースケースを対象とする。

---

## Problem Frame

現行プロジェクト（またはホスト環境）は IPv4 中心の運用を前提としており、双スタックまたは純 IPv6 環境への移行時に、IP アドレス体系、ルーティング、名前解決、ファイアウォール、プロキシ設定などでブロッカーが生じやすい。また、既存のシェルスクリプトとインフラドキュメントは v4 表記で固定されているため、ネットワーク変更を手作業で追記する負荷が高い。

本計画は「追加検証なしに本番適用できる状態」までに整備することを目標とする。評価実験は別環境で実施済みであり、この計画はその知見を反映する。

---

## Requirements

- R1: ホストは IPv4 と IPv6 の両方が動作する（デュアルスタック）か、あるいは純 IPv6 モードのいずれかで起動できる。
- R2: ホスト名とサービス名は IPv4/IPv6 双方の名前解決で到達可能である。
- R3: 既存の IPv4 レンジを前提としたファイアウォール規則は、v6 に対応するレンジへ等価変換され漏れなく適用される。
- R4: インフラスクリプトと設定ファイルは IPv4 固定表記を排除し、IP バージョンによらず同一の手順でデプロイできる。

---

## Key Technical Decisions

- KTD1: アドレス体系とネットワーク設計は `docs/plans/2026-06-06-example-plan.md` に記された方式に従う。
- KTD2: デフォルトはデュアルスタックとし、運用上の理由で純 IPv6 が必要な場合にのみモード切替で対応する。
- KTD3: ファイアウォールポリシーは IP プロトコルを意識したルールセットで管理し、v4/v6 で二重メンテしない。
- KTD4: リバースプロキシはリスナーレベルで両バージョンを受け付ける構成に変更し、アプリケーション層の変更を最小限に抑える。

---

## Implementation Units

### U1: ネットワークと IP アドレス体系の定義

必要なサブネット、アドレス帯、ドメイン、逆引きゾーン、IPAM 記録方式を確定する。v4 と v6 の対応表を作成し、以降の実装で同一スキームを再利用できるようにする。

### U2: OS ネットワークスタックの設定変更

NIC のアドレス付与、ルーティング、ネームリゾルバの設定変更を行う。設定は冪等になるようにスクリプト化し、既存設定との差分を最小にする。

### U3: インフラスクリプトの IP バージョン汎用化

デプロイや初期化で IP アドレスをハードコードしている箇所を洗い出し、変数化または検出ロジックに置き換える。優先度の高いスクリプトから段階的に適用する。

### U4: ファイアウォールとサービス境界の v6 適用

iptables/nftables、あるいはクラウド SG 相当の規則を v6 用に拡張する。禁止・許可ポリシーが v4 と v6 で同一になることを検証する。

### U5: リバースプロキシと L7 ルーティングの修正

リバースプロキシのリスナー追加以降、ホストベースルーティング、 TLS 設定を両バージョン対応にする。証明書と SNI の動作を確認する。

### U6: 確認用テスト項目の作成

疎通、名前解決、証明書、ファイアウォール、プロキシ連携のテストケースを文書化する。v4 専用テストは v6 でも成立するかを明記する。

### U7: オペレーションとトラブルシューティングの整備

変更手順、ロールバック条件、主要な切り分けポイント、よくある原因と対処を整理し、保守担当者が確認できる状態にする。

---

## Test Scenarios

- T1: 新規ホストを dual-stack で起動し、IPv4 と IPv6 の双方からホスト名で到達できる。
- T2: 該当ポートに対して v4 と v6 から接続要求を送信し、許可ポリシーと拒否ポリシーが期待通りに分岐する。
- T3: IPv6 のみの環境でプロキシを経由し、アプリケーションの応答が同じ内容になる。
- T4: 既存スクリプトを修正後に再実行し、複数回実行しても設定が侵食されないことを確認する。
- T5: 証明書の取得リクエストを IPv6 経路で実行し、証明書チェーンが切れないことを確認する。

---

## Risks & Dependencies

- 外部データセンターやクラウド事業者の IPv6 対応範囲は環境依存であり、利用可否については事前の現地調査が必要になる。
- アプリケーションが IPv4 固定ライブラリやハードコーディングされたループバックに依存している場合、ネットワーク層の変更だけでは不足し、アプリ側改修が発生する。
- 運用チームが IPv6 運用の実績を持たない場合、トラブルシュート時の心理的負荷が高く、判断基準の整備が先に必要になる。

---

## Acceptance Examples

- 既存の IPv4 前提の配布手順書を見た運用担当者が、特別な追加調査なしに v6 対応までの手順を完了できる。
- IPv6 のみ有効にしたテスト環境で、主要フローが同一動作をする。
- 緊急時に IPv4 にロールバックする手順が 15 分以内に完了する。
