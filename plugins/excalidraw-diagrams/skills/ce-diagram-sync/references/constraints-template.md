# Architecture Constraints Template

This template defines the schema for `architecture-constraints.md`. Fill in all sections when running `/ce:diagram`. The file is consumed by `/ce:plan` to anchor architectural decisions before implementation.

---

## Template

```markdown
# Architecture Constraints
_Generated by /ce:diagram on {YYYY-MM-DD}. Do not edit manually — use /ce:diagram:update._

## Layer Ownership

| Component | Layer | File/Bloc | Notes |
|-----------|-------|-----------|-------|
| {ComponentName} | {Screen\|Bloc\|Repository\|Service\|Model} | {lib/features/.../...} | {any constraints or empty} |

## Data Flow

| From | To | Direction | Trigger |
|------|----|-----------|---------|
| {ComponentA} | {ComponentB} | {-> \| <- \| <->} | {EventName \| MethodName \| description} |

## Visual References

| Screen | Stitch ID | Applies To |
|--------|-----------|------------|
| {ScreenName} | {PID or URL or n/a} | {component names} |

## Explicit Constraints

- {ComponentName} must NOT {do something}: {rationale}
- {ComponentName} must ALWAYS {do something}: {rationale}
- All {resource type} must come from {source}: {rationale}

## Out of Scope

- {Feature or component this implementation does NOT touch}
- {Behaviour that is intentionally excluded from this iteration}

## Open Questions

- {Unresolved decision that /ce:plan must address before implementing}
- {Ambiguity that needs clarification before work begins}
```

---

## Layer Definitions

Use these exact layer names in the `Layer` column:

| Layer | Meaning | Directory Convention |
|-------|---------|---------------------|
| `Screen` | UI/presentation — Flutter widgets and pages | `lib/features/*/presentation/` |
| `Bloc` | State management — BLoC/Cubit classes | `lib/features/*/bloc/` or `*_bloc.dart` / `*_cubit.dart` |
| `Repository` | Data access — abstracts data sources | `lib/features/*/repository/` or `*_repository.dart` |
| `Service` | External services — APIs, Firebase, device APIs | `lib/core/services/` or `*_service.dart` |
| `Model` | Data models — immutable data classes | `lib/features/*/models/` or `*_model.dart` |

---

## Data Flow Direction Symbols

| Symbol | Meaning |
|--------|---------|
| `->` | A calls/depends on B (A imports B) |
| `<-` | B calls/depends on A (B imports A) |
| `<->` | Bidirectional dependency (use sparingly — usually a design smell) |

---

## Explicit Constraints Format

Write constraints as actionable rules, not vague notes:

**Good:**
- `HomeBloc must NOT handle onboarding state — belongs in OnboardingBloc`
- `All icons must come from Stitch design system, not Flutter Material icons`
- `AuthRepository must ALWAYS be accessed via AuthBloc, never directly from Screen layer`

**Bad:**
- `HomeBloc should be careful about onboarding` (too vague)
- `Use design system` (no specificity)

## Filled Example

```markdown
# Architecture Constraints
_Generated by /ce:diagram on 2026-04-10. Do not edit manually — use /ce:diagram:update._

## Layer Ownership

| Component | Layer | File/Bloc | Notes |
|-----------|-------|-----------|-------|
| HomeScreen | Screen | lib/features/home/presentation/home_screen.dart | Entry point after onboarding |
| HomeBloc | Bloc | lib/features/home/bloc/home_bloc.dart | Owns home feed state only |
| UserRepository | Repository | lib/features/home/repository/user_repository.dart | Fetches user profile |
| AnalyticsService | Service | lib/core/services/analytics_service.dart | Must not be called from Bloc directly |
| UserModel | Model | lib/features/home/models/user_model.dart | Immutable, copyWith pattern |

## Data Flow

| From | To | Direction | Trigger |
|------|----|-----------|---------|
| HomeScreen | HomeBloc | -> | LoadHomeEvent dispatched on init |
| HomeBloc | UserRepository | -> | FetchUserProfile() call |
| UserRepository | AnalyticsService | -> | LogProfileFetch() after success |

## Visual References

| Screen | Stitch ID | Applies To |
|--------|-----------|------------|
| Home Feed | PID-4821 | HomeScreen, HomeBloc |
| Profile Card | PID-4822 | UserModel display |

## Explicit Constraints

- HomeBloc must NOT handle onboarding state — belongs in OnboardingBloc
- AnalyticsService must NOT be called directly from HomeBloc — route through UserRepository
- All icons must come from Stitch design system, not placeholders

## Out of Scope

- Push notification handling
- Offline/cached feed support
- Profile editing

## Open Questions

- Should HomeBloc or a separate FeedBloc own pagination state?
- Are there rate limits on UserRepository.fetchUserProfile() we need to handle?
```
