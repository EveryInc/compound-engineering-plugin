# Excalidraw Diagrams Plugin — Implementation Plan

## Context & Goal

This plugin extends the `compound-engineering` plugin with bidirectional Excalidraw integration.
It solves two specific problems identified in session analysis:
- **30% of redirects** come from architecture placement errors (wrong bloc/layer) — fixed by `/ce:diagram` upfront
- **40% of redirects** come from visual/design spec gaps — fixed by anchoring constraints before `/ce:plan` runs

The plugin is named `excalidraw-diagrams` and lives **alongside** `compound-engineering` in the `plugins/` directory, not inside it. The only modification to the existing plugin is a small hook added to `ce-plan.md`.

---

## Workflow Overview

```
/ce:ideate (optional)
      ↓
/ce:brainstorm
      ↓
/ce:diagram  ✦ NEW
  → if image/file attached: read it → generate architecture-constraints.md
  → if no attachment: scan codebase → generate architecture.excalidraw + architecture-constraints.md
      ↓
/ce:plan  (modified: reads architecture-constraints.md if present)
      ↓
/ce:work
      ↓
/ce:review
      ↓
/ce:diagram:update  ✦ NEW
  → diff what was built vs architecture-constraints.md
  → update architecture.excalidraw
  → append to architecture-changelog.md
      ↓
/ce:compound
      ↓
repeat

On-demand:
/ce:diagram:sync  ✦ NEW  — full rescan when diagram has drifted significantly
```

## Phase 1 — Foundation Files

### File: `references/excalidraw-format.md`

**Purpose:** Teach Claude the Excalidraw JSON schema once, reuse everywhere. Without this, Claude hallucmates element fields and produces invalid files.

**Must cover:**
- Top-level structure: `{ type, version, source, elements, appState, files }`
- Element types: `rectangle`, `ellipse`, `arrow`, `text`, `frame`, `line`
- Required fields per element: `id`, `type`, `x`, `y`, `width`, `height`, `angle`, `strokeColor`, `backgroundColor`, `fillStyle`, `strokeWidth`, `roughness`, `opacity`, `groupIds`, `frameId`, `roundness`, `seed`, `version`, `versionNonce`, `isDeleted`, `boundElements`, `updated`, `link`, `locked`
- Arrow-specific fields: `startBinding`, `endBinding`, `startArrowhead`, `endArrowhead`, `points`
- Binding format: `{ elementId, focus, gap }`
- How IDs work: random alphanumeric strings, must be unique across all elements
- `boundElements` on shapes: array of `{ id, type }` referencing arrows bound to that shape
- `frameId`: set to parent frame's ID when element belongs to a frame, else null
- Text element: `text`, `fontSize`, `fontFamily`, `textAlign`, `verticalAlign`, `baseline`, `containerId` (set when text is label inside a shape)
- Layout conventions: x increases rightward, y increases downward, origin top-left
- Minimal valid element example for each type (rectangle, arrow, text)
- Common mistakes to avoid: mismatched bindings, missing boundElements backlinks, invalid points arrays on arrows

**Implementation note:** Pull reference from `yctimlin/mcp_excalidraw`'s cheatsheet at `skills/excalidraw-skill/references/cheatsheet.md` — ask Claude Code to fetch and summarise it into this file.

### File: `references/constraints-template.md`

**Purpose:** The contract between `/ce:diagram` and `/ce:plan`. A structured markdown template that captures architectural decisions before implementation starts. Gets injected into every `/ce:plan` call automatically.

**Template structure to implement:**

```markdown
# Architecture Constraints
_Generated by /ce:diagram on {date}. Do not edit manually — use /ce:diagram:update._

## Layer Ownership
| Component | Layer | File/Bloc | Notes |
|-----------|-------|-----------|-------|
| {name}    | {UI/Bloc/Repo/Service} | {path} | {any constraints} |

## Data Flow
| From | To | Direction | Trigger |
|------|----|-----------|---------|
| {component} | {component} | {→ / ←} | {event/call} |

## Visual References
| Screen | Stitch ID | Applies To |
|--------|-----------|------------|
| {screen name} | {PID or URL} | {components} |

## Explicit Constraints
- {constraint}: {rationale}
- e.g. "HomeBloc must NOT handle onboarding state — belongs in OnboardingBloc"
- e.g. "All icons must come from Stitch design system, not placeholders"

## Out of Scope
- {what this feature explicitly does NOT touch}

## Open Questions
- {anything unresolved that /ce:plan should flag before implementing}
```

**Implementation note:** Tailor the default column names to Flutter/BLoC architecture (Bloc, Repository, Screen, Service, Model) since that matches the get-fit and mason projects.

### File: `SKILL.md`

**Purpose:** Master context file. Claude reads this at the start of any `/ce:diagram*` command invocation.

**Must cover:**
- Philosophy: diagram is a spec, not documentation. It's written before code, updated after.
- The three project artifacts and their roles
- When to use each command (decision tree)
- Bidirectional workflow: input path (you draw → constraints) vs output path (Claude generates → you review)
- Quality rules for generated diagrams:
  - Max 3 nesting levels
  - Layer ordering: top = UI/Screen, middle = Bloc, lower = Repository, bottom = Service/DB
  - Arrow labels must be event or method names, not generic "calls"
  - Frame per feature area (onboarding, home, profile, etc.)
- How `architecture-constraints.md` is consumed by `/ce:plan` (injected as first context block)
- Changelog format: `## {date} — {command used}\n{bullet list of what changed}`
- When to use `/ce:diagram:sync` vs `/ce:diagram:update`:
  - `:update` = incremental after each review cycle, same feature
  - `:sync` = after major refactor, after 3+ update cycles, when diagram feels stale

## Phase 2 — Agents

### Agent: `agents/diagram-reader.md`

**Purpose:** Reads an attached image (PNG screenshot of hand-drawn diagram or Stitch export) or `.excalidraw` file and produces a populated `architecture-constraints.md`.

**Inputs:**
- Attached file path (image or `.excalidraw`)
- Optional: user description of what the diagram shows

**Output:**
- Populated `architecture-constraints.md` written to project root
- Console summary: list of components identified, flows identified, any ambiguities flagged

**Logic to implement:**
1. If input is image: describe all shapes (boxes, arrows, labels) using vision. Map box labels to Layer Ownership table. Map arrows to Data Flow table. Ask user to confirm any labels that are ambiguous before writing file.
2. If input is `.excalidraw`: parse elements array. Extract all `rectangle`/`ellipse` elements with text labels → Layer Ownership. Extract all `arrow` elements with startBinding/endBinding → Data Flow. Extract all `frame` elements → feature areas.
3. Populate `Visual References` section: prompt user for Stitch IDs for any component that has a visual representation.
4. Populate `Explicit Constraints` section: infer from arrow directions and layer positions. Flag any component that appears to be cross-layer (e.g. UI element directly connected to Repository).
5. Write file. Print summary.

**Error handling:**
- If a shape has no label: flag it as "unnamed component — please clarify before running /ce:plan"
- If an arrow has no clear source or target: flag it
- If constraints file already exists: ask user whether to overwrite or merge

---

### Agent: `agents/diagram-generator.md`

**Purpose:** Scans the codebase and generates a valid `architecture.excalidraw` file representing the current or planned structure. Also produces `architecture-constraints.md`.

**Inputs:**
- Project root (scans file tree)
- Optional: user description of what to include ("show onboarding flow only")
- Optional: existing `architecture-constraints.md` to use as scaffold

**Output:**
- `architecture.excalidraw` written to project root
- `architecture-constraints.md` written/updated in project root

**Logic to implement:**
1. Scan file tree. Identify layers by directory convention:
   - `lib/features/*/presentation/` → Screen layer
   - `lib/features/*/bloc/` or `*_bloc.dart` → Bloc layer
   - `lib/features/*/repository/` or `*_repository.dart` → Repository layer
   - `lib/core/services/` or `*_service.dart` → Service layer
2. Group by feature area (directory name under `lib/features/`).
3. Create one `frame` element per feature area.
4. Within each frame, create `rectangle` elements per component, positioned by layer (top to bottom: Screen → Bloc → Repository → Service).
5. Create `arrow` elements for dependencies: scan import statements to detect which files import which. Arrow direction = dependency direction.
6. Add `text` elements as labels on arrows where import suggests a specific method or event.
7. Write valid Excalidraw JSON using schema from `references/excalidraw-format.md`.
8. Generate `architecture-constraints.md` from the diagram elements.
9. Print: component count, arrow count, feature areas found, any files that couldn't be classified.

**Quality rules:**
- Frame width: 400px per feature area, 60px gap between frames
- Layer Y positions: Screen=50, Bloc=200, Repository=350, Service=500 (relative to frame)
- Component box size: 160px wide × 60px tall
- Arrow style: `strokeWidth: 2`, `startArrowhead: null`, `endArrowhead: "arrow"`, `roughness: 0`
- Colors by layer: Screen=`#dbeafe` (blue-50), Bloc=`#ede9fe` (purple-50), Repository=`#dcfce7` (green-50), Service=`#fef9c3` (yellow-50)

---

### Agent: `agents/diagram-differ.md`

**Purpose:** Compares what was actually built against the `architecture-constraints.md` spec and produces a diff for `/ce:diagram:update` to apply.

**Inputs:**
- `architecture-constraints.md` (the plan)
- Current file tree + git diff (what was built)

**Output:**
- Structured diff: list of additions, removals, renames, layer violations
- Does NOT write files — passes diff to the calling command which applies it

**Logic to implement:**
1. Re-scan file tree using same classification logic as `diagram-generator`.
2. Compare against Layer Ownership table in constraints file:
   - Components in constraints but not in file tree → mark as REMOVED or RENAMED
   - Components in file tree but not in constraints → mark as ADDED
   - Components in wrong layer compared to constraints → mark as LAYER VIOLATION
3. Compare Data Flow table against actual import statements:
   - New imports not in constraints → mark as NEW FLOW
   - Flows in constraints where import no longer exists → mark as REMOVED FLOW
4. Return structured diff object (not prose) for the command to consume.

**Output format:**
```
ADDED: [{component, layer, feature_area}]
REMOVED: [{component}]
RENAMED: [{from, to}]
LAYER_VIOLATION: [{component, expected_layer, actual_layer}]
NEW_FLOW: [{from, to, direction}]
REMOVED_FLOW: [{from, to}]
```

## Phase 3 — Slash Commands

### Command: `commands/ce-diagram.md`

**Trigger:** `/ce:diagram`

**Logic:**

```
1. Check for attached file in current message
   → If .excalidraw or image: invoke diagram-reader agent
   → If no attachment: invoke diagram-generator agent

2. Print constraints summary to terminal for human review:
   "Here's what I captured. Confirm before running /ce:plan:"
   → List all components with their assigned layers
   → List all data flows
   → List any open questions

3. Ask: "Does this match your intent? If yes, run /ce:plan."

4. Do NOT invoke /ce:plan automatically — human confirms first.
```

**Flags to support:**
- `--feature [name]` — scope generator to one feature area only
- `--overwrite` — skip confirmation prompt if constraints file already exists

---

### Command: `commands/ce-diagram-update.md`

**Trigger:** `/ce:diagram:update`

**Logic:**

```
1. Invoke diagram-differ agent with:
   → architecture-constraints.md
   → current file tree
   → git diff HEAD (if git is available)

2. If diff is empty: print "Diagram is up to date. No changes needed." and exit.

3. Print diff summary:
   "I found the following changes since the last diagram update:"
   → list each category (ADDED, REMOVED, etc.)

4. Ask: "Apply these changes to architecture.excalidraw and log them?"

5. On confirmation:
   → Apply diffs to architecture.excalidraw
   → Update architecture-constraints.md Layer Ownership and Data Flow tables
   → Append to architecture-changelog.md:
     "## {ISO date} — /ce:diagram:update
     - {bullet per change}"

6. Print: "Diagram updated. {N} changes applied."
```

---

### Command: `commands/ce-diagram-sync.md`

**Trigger:** `/ce:diagram:sync`

**Logic:**

```
1. Warn: "This will regenerate architecture.excalidraw from scratch.
   The existing diagram will be archived as architecture.excalidraw.bak.
   Continue? (y/n)"

2. On confirmation:
   → Archive existing architecture.excalidraw → architecture.excalidraw.bak
   → Invoke diagram-generator agent (full scan, no scaffold)
   → Overwrite architecture.excalidraw and architecture-constraints.md
   → Append to architecture-changelog.md:
     "## {ISO date} — /ce:diagram:sync (full regeneration)
     - Previous diagram archived to architecture.excalidraw.bak
     - {component count} components found across {feature area count} feature areas"

3. Print: "Sync complete. Open architecture.excalidraw to review."
```

## Phase 4 — Integration Hook in `/ce:plan`

**This is the only change to the existing `compound-engineering` plugin.**

In `plugins/compound-engineering/commands/ce-plan.md`, add at the top of the command logic:

```
Before generating the technical plan, check for architecture-constraints.md in the project root.

If found:
  → Read architecture-constraints.md
  → Prepend its full contents as the first block of context under the heading:
    "## Architecture Constraints (from /ce:diagram)"
  → Honour all explicit constraints listed. If the plan would violate any constraint,
    flag it explicitly: "⚠ This would violate constraint: [constraint text]. Confirm?"
  → Reference Stitch IDs from the Visual References section when describing UI work.

If not found:
  → Print: "No architecture-constraints.md found. Consider running /ce:diagram first
    to lock in architectural decisions before planning. Proceeding without constraints."
  → Continue with normal plan generation.
```

## Notes for Claude Code

- Do not use any external MCP server or Docker dependency. All file operations use standard bash and file tools.
- The `.excalidraw` format is plain JSON — read and write with standard file I/O.
- When generating Excalidraw JSON, always validate: every arrow's `startBinding.elementId` and `endBinding.elementId` must reference an existing element ID. Every element in an arrow's `boundElements` must exist.
- The constraints file is markdown, not JSON — keep it human-readable and editable.
- Do not auto-run `/ce:plan` from within `/ce:diagram` — always pause for human confirmation after showing the constraints summary.
- Follow the same agent/command markdown format used by existing compound-engineering commands.
