# Scoping Dialogue

Scope the idea before building, using a compact PM-facing dialogue. This mirrors `ce-brainstorm`'s discipline as a pattern — it does not invoke `ce-brainstorm`, and this skill references no files outside its own directory.

## Discipline

- **One question at a time**, with the blocking-question tool. Stacking questions dilutes answers.
- **Prefer single-select options** when there are natural choices; leave room for a free-text answer.
- **Start broad, then narrow.** Problem and who has it first; then the core outcome; then constraints and explicit exclusions.
- **Be a thinking partner**, not just an interviewer — offer a sharper framing or a smaller version when you see one.

## What to establish

Enough that the prototype and PRD won't have to invent product behavior:

- **Problem and beneficiary** — whose problem, and what changes for them when this exists. Push past an abstract persona to a specific one.
- **Core outcome** — the single thing the prototype must make real. This is the first slice Phase 4 builds.
- **Primary flow** — the main path a user takes, start to finish.
- **Constraints and non-goals** — what's explicitly out, so scope creep is visible later.
- **What "good" looks like** — the observable signal the idea is working (this feeds the measurability principle in Phase 5).

## Rigor probes

Before moving to build, probe the gaps that would otherwise surface late:
- **Evidence** — the most concrete thing anyone has already done about this problem (paid, built a workaround, complained).
- **Counterfactual** — what the beneficiary does today when this problem arises, and what that costs them.
- **Smallest version** — the smallest prototype that would still prove the idea worth building.

Capture unresolved uncertainty as an explicit assumption rather than skipping it. Carry every scoping decision into the decision log (see `references/decision-log.md`) so it flows to the PRD.
